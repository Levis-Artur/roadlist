# Електронний маршрутний лист

MVP PWA-застосунку для початку та завершення зміни патрульного екіпажу. Дані зберігаються локально в `localStorage`; авторизація, backend та OCR замінені тестовою логікою.

Після перевірки жетона MVP підтягує тільки ПІБ, номер жетона та УПП. Звання працівника не використовується.

Текстові дані маршрутних листів зберігаються в `localStorage`. Фото одометра стискаються до JPEG (максимум 1280×1280, якість 0.7) і зберігаються окремо в IndexedDB `route-sheet-photo-db`; маршрутний лист містить лише ідентифікатори фото.

## Запуск

```bash
npm install
npm run dev
```

Відкрийте адресу, яку покаже Vite (типово `http://localhost:5173`). Для production-збірки:

```bash
npm run build
npm run preview
```

## Тестові жетони

- `0000001` / PIN `1111` — Іваненко Іван Іванович
- `0000002` / PIN `2222` — Петренко Петро Петрович
- `0000003` / PIN `3333` — Сидоренко Андрій Миколайович

Номер жетона завжди зберігається як рядок із рівно 7 цифр (`^\d{7}$`), щоб не втрачати початкові нулі. Старі тестові номери `000001`, `000002`, `000003` більше не використовуються для нових операцій.

Патрульний входить за номером жетона та PIN із 4–8 цифр. Backend зберігає тільки bcrypt hash PIN і видає JWT на 12 годин; start/finish беруть ідентичність патрульного з токена, а не з body. У довіднику «Користувачі» PIN задається під час створення і може бути замінений під час редагування, але ніколи не показується чи не експортується.

Mock OCR повертає `198234` на початку та `198376` наприкінці зміни. Для перевірки інших значень скористайтеся кнопкою «Внести вручну».

Поле «Номер екіпажу / підрозділу» є необов’язковим. Його можна використовувати для номера екіпажу, групи, сектору або підрозділу; якщо така нумерація не використовується, поле залишають порожнім. У базі порожнє значення зберігається як `null`, а в таблицях і звітах відображається як `—`.

## Адміністративна панель

Адмін-панель відкривається тільки за адресою `/admin`. Вхід виконується за логіном і паролем через backend `POST /api/admin/login`; у `sessionStorage` зберігаються JWT і публічні дані адміністратора.

Тестовий системний власник створюється seed-ом:

```text
owner / owner12345
```

> Пароль `owner12345` призначений лише для локального MVP/seed. Seed створює owner з прапорцем `mustChangePassword`, тому після першого входу система примусово попросить встановити власний надійний пароль.

> Дані залишаються лише в поточному браузері. Очищення даних сайту видалить маршрутні листи та фотографії.

## Архітектура MVP

- React PWA з окремими маршрутами патрульного `/` та адміністратора `/admin`.
- `officerService` виконує вхід за жетоном і PIN та зберігає сесію в `sessionStorage`; mock-реалізація лишається fallback для локального MVP.
- Тільки текстові дані маршрутних листів зберігаються в `localStorage`.
- Стиснені фото одометра зберігаються в IndexedDB окремо від маршрутних листів і мають строк зберігання 30 днів.
- Компоненти працюють через service layer і не звертаються напряму до mock-даних, `localStorage`, IndexedDB або `sessionStorage`.
- `ocrService` імітує майбутній OCR API, а `adminService` працює з backend JWT-авторизацією адміністраторів.
- Ключові операції записуються в локальний журнал дій.

Service layer підключений до backend API; локальні реалізації використовуються лише як fallback при мережевій недоступності сервера.

## Backend MVP

Окремий Express + TypeScript + Prisma backend розміщений у [`server`](./server). Інструкції запуску, Docker Compose і приклади endpoint-ів наведені в [`server/README.md`](./server/README.md).

## Розгортання на Debian

Production-like запуск через Docker Compose (nginx + frontend, backend і PostgreSQL) описаний у [`DEPLOY_DEBIAN.md`](./DEPLOY_DEBIAN.md). Назовні публікується лише порт `80`; backend і база даних доступні тільки всередині Docker network.

## Підключення до backend

Frontend використовує `VITE_API_URL` із `.env` і типово очікує API на `http://localhost:4000`. Приклад конфігурації наведено в [`.env.example`](./.env.example).

```bash
# backend
cd server
cp .env.example .env
docker compose up -d
npm install
npm run prisma:generate
npm run prisma:migrate -- --name init
npm run prisma:seed
npm run dev

# frontend (в іншому терміналі, з кореня проєкту)
npm install
npm run dev
```

Officer verification, route sheets, фото, OCR, admin login та audit logs спочатку звертаються до backend API. Локальні mock/`localStorage`/IndexedDB залишені лише як fallback для frontend MVP, якщо API мережево недоступне; адміністративний вхід потребує backend.

> JWT і PIN забезпечують базову авторизацію, але не замінюють повний production-контроль доступу. У production потрібно задати сильний `JWT_SECRET`, використовувати HTTPS, безпечно керувати сесіями та змінити seed-паролі.

Для публічного доступу обов’язково використовуйте HTTPS. Для закритого тесту можна використовувати Tailscale, але не використовуйте систему з персональними даними через відкритий HTTP у публічному інтернеті. Не відкривайте PostgreSQL назовні і не публікуйте backend port `4000`; API має бути доступний лише через захищений reverse proxy.

## Ролі доступу

`SYSTEM_OWNER` — один власник системи. Створюється seed-ом, бачить усі УПП, може створювати `NATIONAL_ADMIN` і `REGIONAL_ADMIN`. Його не можна редагувати, деактивувати або створити повторно через адміністративну панель.

`NATIONAL_ADMIN` — національний адміністратор. Бачить дані по всій Україні, може створювати тільки `REGIONAL_ADMIN`, перевіряти маршрутні листи, закривати місяці, керувати патрульними й автомобілями.

`REGIONAL_ADMIN` — адміністратор конкретного УПП. Backend обмежує його доступ тільки власним `department`: патрульні, автомобілі, маршрутні листи, місячні листи, журнал дій і статистика фільтруються по УПП. Вкладка «Адміністратори» для цієї ролі недоступна.

Адміністраторські паролі зберігаються тільки як bcrypt hash. Нові адміністратори отримують тимчасовий пароль і зобов’язані змінити його перед доступом до адмінки. Вимоги до нового пароля: мінімум 12 символів, велика і мала літера, цифра та спецсимвол. Після 5 неправильних спроб входу акаунт тимчасово блокується на 15 хвилин. Admin JWT живе `ADMIN_JWT_EXPIRES_IN` (типово 2 години), officer JWT — `OFFICER_JWT_EXPIRES_IN` (типово 12 годин).

## Двофакторна автентифікація

Для всіх адміністративних ролей 2FA через Google Authenticator / TOTP є обов’язковою. Адміністратор спочатку вводить логін і пароль, після чого система або просить змінити тимчасовий пароль, або показує QR-код для першого налаштування 2FA, або просить 6-значний код із застосунку.

Повний admin JWT видається тільки після успішної перевірки TOTP. До цього використовується короткий `admin_2fa_pending_token`, який дозволяє лише змінити тимчасовий пароль, налаштувати 2FA або підтвердити код.

При втраті телефона 2FA може скинути тільки `SYSTEM_OWNER` для `NATIONAL_ADMIN` або `REGIONAL_ADMIN`. Секрет TOTP не показується після активації, не експортується і не пишеться в logs.

## Загальна логіка

Адміністратор додає патрульних та службові автомобілі у відповідних довідниках. Активний патрульний може увійти за жетоном і PIN, а всі активні автомобілі доступні для маршрутного листа. Якщо активний автомобіль один, він підставляється автоматично; якщо їх декілька, патрульний обирає потрібний зі списку.

## Керування патрульними

Після входу в `/admin` відкрийте вкладку «Користувачі». Тут можна додати або відредагувати ПІБ, номер жетона, УПП, PIN та активність, відфільтрувати довідник і експортувати `officers-export.csv`. Дія «Деактивувати» не видаляє запис: історичні маршрутні листи залишаються доступними.

## Керування автомобілями

У вкладці «Автомобілі» доступні додавання, редагування, фільтри, CSV і soft-disable. Адміністратор вводить номер для відображення, наприклад `АА5200МН`, а система автоматично зберігає нормалізований `AA5200MH`: прибирає пробіли, переводить символи у верхній регістр і замінює схожі українські літери латинськими.
